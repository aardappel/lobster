Lobster Comparative Documentation Generator (CDG) Tool Guide
Introduction

Hello! Thanks for taking the time to look at my documentation generator. I hope to add more features to it (and make it more stable) as Lobster grows as a language. Anyways, why was this made? I found Lobster via Hacker News and thought it was a really cool language. I decided to try it out and mess around with it like I would with any language. I was checking the "C-style language cheat sheet" often and noticed it was somewhat outdated (dynamic scoping being removed, operators like :=, :==, <- gone too, etc.). I decided I would try to update the cheat sheet and contribute, that way anyone else trying Lobster out for the first time wouldn't run into any pitfalls like I did. After checking the "/docs/source/" directory for any file for the cheat sheet, then realizing there wasn't one, I tried manually editing the source of the cheat sheet html file (bad idea). It looked like it was generated by a program, so I thought "Why don't I make something that writes the cheat sheet, in Lobster??" - So I wrote something that generates cheat sheet-like files (in Lobster). While I hope you can get some use out of this tool besides writing the cheat sheet, I also realize that this tool's scope is a bit limited. But that's alright with me as I learned a lot about Lobster, and I like contributing to open source projects.

Basics

"Doc files" use a very basic syntax. This section will cover everything you need to know to properly generate comparative documentation files.

CDG only requires one file to be present in the "/docs/source/" directory. This file (for now) is locked at "new_doc.doc". In the future you will be able to specify your own file to CDG. The first 2 lines of the file should contain the following properties:

title: [title of your documentation]

output: [output filename (without extension)]

CDG documents are built up of 3 parts: Headers, Subtext, and Examples. Headers are denoted with the ! character at the beginning of a line. Headers can be followed immediately by Subtext or any number of Examples. For Subtext, simply write as you would normally below the Header line. For now, Subtext gets parsed directly into HTML, so tags such as <em>, <strong>, etc. will work. Headers require at least 1 Example, but do not require Subtext.

Examples can be written like so:

 # Indendation-based syntax with an inline function definition.
 # "this" is inferred automatically.
 `
 struct xy:
   x:float
   y:float
   def magnitude():
     sqrt(x * x + y * y)
 `
 `
 public class xy {
   float x;
   float y;
   float magnitude() {
     return x * x + y * y;
   }
 }
 `
As you can see above, Examples have 3 main pieces:
  • The description (denoted by the # character)
  • The comparator (written between ` characters)
  • The comparison (also written between ` characters)
Each of these pieces can span multiple lines. However, the description needs a # character starting each line it extends to. Comparators and comparisons do not. Descriptions work similar to Subtext, in that HTML elements are included in the final output.

Here's a small example of what a standard CDG file could look like:
 title: Lobster to JavaScript Comparison
 output: lobster_to_javascript
 
 ! Assigning Functions
 Assigning function values in Lobster works similarly to JavaScript[...] 
 
 # Pre ES6 syntax
 ` let add = def(a, b): return a + b `
 ` var add = function(a, b) { return a + b }; `
 
 # Post ES6 syntax
 ` let add = def(a, b): return a + b `
 ` let add = (a, b) => { return a + b }; `

 [...]
